# AWM2 / Yamaha MU100 emulation — design

Date: 2026-06-13
Status: design agreed; spike not yet started

## Goal

Build a `schwung-awm2` module that reproduces Yamaha AWM2 synthesis by emulating
the **MU100** sound module (Hitachi H8S/2655 CPU running the real firmware +
Yamaha **SWP30** AWM2/MEG chip), driven by MIDI. The payoff over a soundfont is
the MEG effects engine (XG reverb/chorus/**variation**) and the complete XG
spec provided by the firmware — neither of which can be reproduced by a
soundfont (the wave ROM samples alone *can*, and already exist as the DB50XG
SF2, which is why MU50/SWP00 was rejected as too close to "just a soundfont").

## Key decisions

1. **Target = MU100 / SWP30** (not MU50/SWP00, not MU80/SWP20). Highest payoff:
   the rich XG variation effects that "can't be soundfonted."
2. **No DRC.** `swp30_device::execute_run()` branches on `m_meg_drc_active`;
   when false it runs the MEG microcode via the `meg_state::step()` interpreter
   (swp30.cpp:3743). Force the flag false → the entire `cpu/drcuml` dependency
   is removed. This eliminates the scariest part of the port.
3. **Whole-module emulation** (H8S/2655 + SWP30 + firmware), feed MIDI into the
   H8 SCI rx — same approach as the JV-880 module. Driving SWP30 registers
   directly was rejected: the firmware *is* the synth (voice alloc, XG spec).
4. **Spike-first.** Prove faithfulness + measure CPU cost OFF-DEVICE before any
   Move integration or vendor-vs-shim framework decision.
5. **Build the spike on the `ensoniq-sd1-source` patched-MAME tree**, not a
   fresh upstream checkout. It already compiles a headless-ish MAME core on
   macOS with a custom `emu.h` + `Mame_Patches/`. Swap the Ensoniq machine for
   the MU100. (Risk: MAME version drift between that tree and the swp30/h8
   files pulled from master — must reconcile APIs.)

## Oracle / faithfulness test

Use **stock upstream MAME** as ground truth: run `mame mu100` headless with a
test MIDI file → `ref.wav`. Our extraction is "correct" when it produces
(near-)identical samples for the same ROMs + MIDI → `out.wav`. Turns subjective
"sounds right" into a diffable test.

## ROMs (user-supplied, verified 2026-06-13)

The user obtained the `mu100` romset locally; all CRC32s verified against the
`ymmu100.cpp` driver manifest. **Engine ships without ROMs; user supplies them**
(same posture as JV-880).

Program ROM (region `maincpu`): `xu50720.ic11` 2MB `1126a8a4` (v1.11 default).
Wave ROM (region `swp30`, ~24MB):
`xs518b0.ic34` `2550d44f` · `xs743b0.ic35` `a9109a6c` ·
`xt445a0-828.ic36` `225c2280` · `xt461a0-829.ic37` `a1d138a3` ·
`xt462a0.ic39` `2e82cbd4` · `xt463a0.ic38` `cce5f8d3`.
Device ROMs: `sin-table.bin` `4305f63c` (SWP30 internal quarter-sine, 64KB,
MAME-generated not Yamaha); `hd44780u_b04.bin` `126ed6da` (LCD charset; display
will be stubbed but device init may want the region).

## Extraction surface (from MAME master)

- `src/devices/sound/swp30.{cpp,h}` (+ `meg`, `dirom` deps) — staged in
  `../mame-swp30-ref/`.
- `src/devices/cpu/h8/*` — h8 base + `h8s2000`/`h8s2600`/`h8s2655` chain +
  peripherals (`h8_adc`, `h8_intc`, `h8_port`, `h8_sci`, `h8_timer8/16`,
  `h8_watchdog`, `h8_dma`, `h8_dtc`) + disassembler. Instruction tables are
  generated by `h8make.py` from `h8.lst`.
- `src/mame/yamaha/ymmu100.cpp` + `mulcd` (stub) + `adc` + MIDI bus.

## Spike harness

Headless `main`: instantiate mu100 machine → load ROMs → feed `.mid`/raw MIDI
into H8 SCI rx → run scheduler → write stereo `out.wav`. No render/ui/lua.
Then diff vs stock-MAME `ref.wav`. Measure CPU cost (cycles per second of
audio) to predict Move ARM feasibility.

## Open / deferred

- Framework strategy for the actual Move port (vendor MAME core headless vs.
  surgical clean-room `emu.h` shim) — **decided after the spike** based on
  measured CPU cost. No one has run MAME-core on the Move yet; existing Move
  emulator modules (JV-880, Osirus, Surge) are focused extractions, not MAME.
- Module packaging, UI/params, XG part mapping to Move controls.
